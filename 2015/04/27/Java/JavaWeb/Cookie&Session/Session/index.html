    • 当第一次请求该Servlet资源时，没有任何Cookie，此时request.getSession() 会创建一个新的HttpSession对象，而这个HttpSession对象是通过 Cookie技术实现的。它的name属性为JSESSIONID，value值为一个很长的数组字母组合。 而且响应头中写入一个Cookie，它大致为： Set-Cookie: JSESSIONID=85DF4EE96EE132E8736748CCC6597625; Path=/session01/; HttpOnly；

    • 第二次访问时，就能获取到Cookie且name为JSESSIONID，这时的请求头： Cookie: JSESSIONID=85DF4EE96EE132E8736748CCC6597625， 而响应头这时是没有Cookie的。后续继续访问该资源都是请求头携带Cookie，而响应头没有Cookie，直到会话结束；

    • 当关闭浏览器（退出进程），重新打开新的浏览器访问资源时，也是没有Cookie！这说明了上一次会话Session对应的Cookie是一个临时的Cookie（从输出内容可以看到maxAge=-1），所以才会在浏览器退出是被删除。

    注意：
    1）这个Session（Cookie）的存活时间就是前面说的一次会话，即浏览器关闭后， 下次打开浏览器访问会创建新的Session（对应Cookie的值会不一样）。
    2）不同用户或者同一个用户的不同浏览器之间调用getSession创建的Cookie是不同的，这可以从Cookie的值上看得出。 而打开同一个浏览器的不同窗口或者多个同款浏览器的进程之间的Session对应的Cookie是相同的。

    结论

    第一次访问时，没有携带JSESSIONID，此时服务端直接创建一个HttpSession对象，并且以响应头Set-Cookie的方式设置到浏览器端；第二次访问时就携带了JSESSIONID，服务端根据这个JSESSIONID知道这是已经创建了的HttpSession对象，所以直接返回该对象。

    当关闭浏览器后，浏览器端的Cookie会被清除，此时再次访问的话就不会携带JSESSIONID，服务器端就认为这是个新的请求，重新创建HttpSession对象；而服务器应该维护着大量的HttpSession对象，并在合适的时间销毁。

    Returns the current session associated with this request, or if the request does not have a session, creates one.

    Session失效问题

    前面说明的Session的原理后，知道是通过Cookie技术实现的。而Cookie可以设置存活时间，那么默认的Session时间是30分钟， 也就是浏览器没关，等待30分钟后Session也会失效，再次访问时就会调用getSession(true)方法的话创建新的Session。

    注意，这里所说的失效是针对服务端来说的！因为在客户端只要关闭了浏览器进程Session就会销毁！

    如果没到达30分钟就关闭浏览器后再重新访问，这时Session是新创建的，而原先服务器端的Session其实也还存在于内存中，但已经不能获取了！

    说明

    1）优先级：setMaxInactiveInterval > 应用的web.xml设置 > Tomcat的web.xml设置。
    2）若访问服务器session超时，即上次会话结束，但服务器与客户端会产生一个新的会话，之前的session里的属性值全部丢失。
    3）客户端与服务器一次有效会话（session没有超时），若代码中设置了setMaxInactiveInterval值，那么这个session的最大不活动间隔时间将被修改，并被应用为新值。
    4）Session在这些情况销毁：某个请求周期内调用了Session.invalidate()方法，此请求周期结束后，session被销毁；
    或者是session超时后自动销毁；或者客户端关掉浏览器。
    5.对于JSP，如果指定了<%@ page session="false"%>，则在JSP中无法直接访问内置的session变量，同时也不会主动创建session，
    因为此时JSP未自动执行request.getSession()操作获取session。

    上面修改Session有效时间和获取Session对应的Cookie对象，然后用setMaxAge修改存活时间的方法有什么区别？？？？
    个人思考，正确性未知：
    应该避免这种用法，因为设置Cookie的存活时间是客户端计算的，而Session的有效时间又是服务端计算的，两者的时间不统一，
    很难知道Session是否已经失效或者Cookie是否存活在下次请求头中。
    另一种说法：
    如果通过创建一个名为 JSESSIONID的Cookie保存在客户端中并设置了生命周期为最大，那么当服务器端失效后，
    客户端的Cookie依然存在，但传递过去后由于服务器端已经没有对应的HttpSession，所以不能识别。

    HttpSession对象的搁置与激活

    会话使用时可能出现的现象
    1）HttpSession对象可能会很多，内存不够
    2）用户离开电脑时间比较长(但 <30 分钟)
    3）应用重新启动了(对于用户购买的商品应该还在)

    出现这些情况的时候，可以把HttpSession 对象持久化到硬盘中(利用的是对象的序列号操作)，
    等到需要的时候重新从硬盘中反序列化就能获取原来的对象。

    问题补充：

    1）第一方Cookie和第三方Cookie
    第一方cookie指的是访客当前访问的网站给访客的浏览器设置的cookie, 会被存储在访客的计算机上。
    第三方cookie指的是当前访问的网站中会加载另外第三方的代码，例如促销广告，那么第三方也会在访客的计算机上添加cookie，这种就是第三方cookie。例如访问新浪微博，京东的广告出现了，那么京东有可能就会在访客的计算器上添加这个cookie文件, 京东就属于第三方。
