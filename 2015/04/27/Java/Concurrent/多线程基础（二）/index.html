1）等待唤醒机制

对于调用wait、notify和notifyAll的方法，都只能在同步代码块或同步函数中使用，而且调用这些方法的对象应该是线程的锁对象（监视器） 。
因此在同步代码块中，就应该由锁对象调用；而在同步函数中，则是当前对象调用（即this对象）；而在静态同步函数中则应该由Class对象调用。

下面是JDK 文档中对wait方法的说明：
The current thread must own this object’s monitor. The thread releases ownership of this monitor and waits until another thread notifies threads waiting on this object’s monitor to wake up either through a call to the notify method or the notifyAll method. The thread then waits until it can re-obtain ownership of the monitor and resumes execution.
This method should always be used in a loop:

This method should only be called by a thread that is the owner of this object’s monitor.

从上面知道，wait方法执行时会释放线程持有的锁对象。

wait方法再次说明

某个线程试图去执行某个实例的wait方法时，这个线程必须获取该实例的锁定。然而当程序执行流到达wait方法的代码后，线程会释放锁并且等待，这时候别的线程就能获得锁了。等待的线程可能被别的线程执行notify、notifyAll或interrupt退出。最后退出等待进入下一个语句之前必须再次获取锁定。

出现上面所说的问题主要是：

  • 在判断标志位flag的时候用的是if
    假如flag为false，此时有一个消费者线程获取了锁，根据判断该线程执行wait方法等待并释放锁；接着又有一个消费者线程获得了锁，同样该线程执行wait方法并等待。
    这时候有一个生产者线程获取了锁，执行了wait后面的代码，增加了一个产品，并把 flag设置为true，同时唤醒其中一个消费者线程。
    该消费者线程被唤醒并获得了锁，从if判断结构出来执行后面的代码，减少一个产品，并把flag设置为false同时唤醒了另一个消费者线程。
    此时另一个消费者线程也直接从if判断结构出来，也就是没有判断flag！！这样它也减少了一个产品并设置flag为false和唤醒一个其它线程。这样就出现生产一个产品消费两个的情况。
    这个问题的原因是消费者线程被唤醒后没有再次判断flag的值（flag代表是否有产品）。
    所以 正确的方法是使用while进行判断
  • 用while循环判断取代if判断后，还会出现一个问题，那就是线程有可能都处于等待状态。这是因为唤醒线程是使用的是notify，
    而这种情况下唤醒的线程没有保证，很有可能造成生产者只唤醒另一个生产者，消费者也只唤醒另一个消费者的情况。
    所以应该使用notifyAll方法进行唤醒。
